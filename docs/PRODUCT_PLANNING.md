# 产品规划

## 总原则

所有产品规划都按照中远期完全体来描述，不能只按照当前原型能力来命名、解释或设计。

当前实现可以很小，但规划文档必须回答完整系统最终应该长什么样。阶段性实现只是通往完全体的切片，不代表最终产品边界。

## 规划和实现的关系

- 产品规划：面向中远期完全体，描述最终角色、入口、业务流程、权限边界和审计要求。
- 产品路线图：把完全体拆成可验证阶段，见 `docs/ROADMAP.md`。
- 阶段实现：面向当前最小可验证闭环，优先选择成本低、可审查、可迁移的实现方式。
- 文档表达：必须同时写清楚“最终形态”和“当前阶段”，避免把临时工具误认为最终产品。
- 命名约束：如果当前能力只是临时实现，应明确写作“当前阶段使用”，不要命名成最终产品本身。

## 中远期完全体角色

Live OS 至少面向以下角色提供产品能力：

- 普通观察者：查看公开状态、事件流、资源概况、规则版本和系统健康度。
- 候选成员：查看可参与任务、提交劳动、查看自己的贡献记录和状态变化。
- 正式成员：参与任务、资源协作、申诉、规则反馈和社区治理流程。
- 财务治理成员：审核报销、记录付款、维护公开财务透明度。
- 管理员：维护成员、任务、资源、规则、申诉和异常数据，处理日常运营事务。
- 治理成员：查看审计线索，处理高影响争议、规则变更和人工裁决。
- 计划负责人：维护项目执行计划、节点依赖、成本、人力、工期、完成标准和容量影响。
- Simulation Engine：作为外部客户端通过 Live OS API 读取状态、提交行为和接收事件。

## 中远期完全体入口

最终产品不应只有 Django Admin。完整系统应至少包含：

- 观察台：面向公众或观察者的只读入口，展示社区整体运行状态。
- 公开财务：面向公众或观察者的只读入口，展示报销、审核、付款和流水摘要。
- 成员工作台：面向候选成员和正式成员，承载任务领取、劳动提交、贡献记录和个人状态。
- 运营后台：面向管理员，围绕业务流程组织任务、成员、资源、申诉和事件处理。
- 治理后台：面向治理成员，提供规则版本、争议裁决、审计轨迹和高风险操作复核。
- 计划后台：面向计划负责人和治理成员，维护从招募、抵达、建设、扩容到对外服务的完整执行计划。
- API：面向 Simulation Engine 和未来客户端，所有状态变化必须通过契约化 API。
- 仿真实验后台：面向开发者和未来 Simulation Engine，负责启动和推进可复盘的模拟生活；观察台只展示结果。
- 内部维护后台：保留 Django Admin 作为开发、排障和底层数据维护工具。

## Django Admin 的定位

当前阶段 `/admin/` 是管理员可以使用的中文界面，也是最快的本地预览和数据维护入口。

但在完全体规划中，Django Admin 不是最终运营后台。它的定位是内部维护后台，适合：

- 开发期检查模型数据。
- 本地演示和排障。
- 少量低频的底层数据修正。
- 超级管理员在明确知道表结构含义时进行维护。

它不适合承载最终高频业务流程，因为：

- 页面按数据模型组织，不按真实工作流组织。
- 容易暴露底层字段和危险操作。
- 难以表达复杂权限、复核、审计和跨对象流程。
- 不适合作为普通成员、治理成员或观察者的产品体验。

## 运营后台的最终目标

中远期运营后台应围绕流程，而不是围绕数据库表：

- 成员管理：身份状态、画像、贡献历史、风险提示和人工备注。
- 任务运营：发布任务、审核任务、指派或开放领取、关闭未开始任务、验收劳动、触发积分。
- 计划运营：维护主线节点、依赖、预算、人力、工期、完成标准和容量影响。
- 资源管理：查看库存、维护资源状态、处置资源预警、记录资源变动事件。
- 财务管理：处理成员报销、财务审核、付款流水和公开财务说明。
- 申诉处理：查看上下文、记录处理意见、形成可审计结果。
- 规则管理：查看规则版本、准备变更、发布生效记录。
- 事件审计：追踪关键状态变化和责任人。

## 当前阶段策略

当前阶段可以继续强化 Django Admin，因为它能快速服务开发和验证：

- 中文字段和模型名称。
- 列表展示、搜索、筛选和只读字段。
- seed 数据和 smoke workflow。
- 文档化每个模型的用途和危险操作。

当前任务、资源、公开财务和申诉流程优先通过 API 与领域服务承载：`core.tasks.*` 负责任务创建、发布、指派、关闭和验收，`core.resource_services` 负责库存调整和资源事件，`core.finance_services` 负责报销提交、审核、付款和撤回，`core.dispute_services` 负责申诉受理、处理结论和申诉事件，避免管理员为了业务动作直接编辑任务、资源、报销、申诉、积分流水和事件等底层表。

当前仿真控制入口已收敛到 `/admin/simulation-lab/`，固定 world 的 `/` 只负责观察复盘目标 world。每次仿真推进必须归属于 simulation run，并能被观察台看到；不能默认改写真世界任务、库存、积分或计划。

当前新增数据库化项目执行计划。`bigapple001据点执行计划` 不是一次性剧情文本，而是可编辑、可版本化、可被多次模拟引用的真实项目计划源头。它应长期沉淀为真实世界执行计划的参考依据。

但所有相关文档都必须说明：这是当前阶段的内部维护入口，不是最终运营后台。

当以下条件出现时，应启动自定义运营后台：

- 管理员需要频繁跨多个模型完成一个业务动作。
- 某些字段不能暴露给普通管理员直接编辑。
- 业务操作需要复核、审批或明确审计链。
- Django Admin 页面难以解释真实工作流。
- 成员、治理成员或外部观察者需要专属产品体验。
