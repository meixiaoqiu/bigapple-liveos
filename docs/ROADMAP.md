# 产品路线图

## 目的

这份路线图把 Live OS 的中远期完全体拆成可执行阶段。它不以当前 Django Admin 原型为产品边界，而是从最终系统反推当前每一步应该验证什么。

路线图用于回答三个问题：

- 完整系统最终需要哪些产品入口、角色和能力。
- 每个阶段应该交付什么，不应该提前做什么。
- 每个阶段完成后，文档、契约、数据、测试和演示应达到什么标准。

## 路线图原则

- 完全体优先：先定义最终角色、入口、流程和权威边界，再决定当前阶段实现切片。
- 契约先行：面向客户端、Simulation Engine 或未来前端的接口变化，必须先进入 `big-apple-contracts`。
- 权威单一：Live OS 是业务状态、积分账本、规则版本和事件流的唯一权威来源。
- 人机同路：真实成员和 Simulation Engine 必须使用同一组 Live OS API，不允许 Simulation Engine 直接写业务表。
- 审计默认存在：涉及身份、积分、任务验收、资源、申诉、规则和治理的状态变化都应留下事件或审计线索。
- 阶段可验证：每个阶段都要有 seed 数据、测试命令、手工预览入口和文档说明。

## 阶段总览

| 阶段 | 主题 | 主要目标 | 产品入口 |
| --- | --- | --- | --- |
| P0 | 权威原型 | 建立数据模型、API 闭环、中文 Admin、观察台和本地开发流程。 | `/admin/`、`/observer/`、`/api/v0.1/` |
| P1 | 内部维护后台强化 | 让 Django Admin 成为可靠的开发期维护工具，而不是最终运营后台。 | `/admin/` |
| P2 | 成员工作台 | 让候选成员和正式成员能完成任务领取、提交劳动和查看贡献。 | `/workspace/`、API |
| P3 | 运营后台 | 让管理员按业务流程管理成员、任务、资源、申诉和事件。 | API、control `/admin/`、后续专用业务页面 |
| P4 | 治理后台 | 支持规则版本、争议裁决、复核、审计和高影响操作追踪。 | 治理后台、API |
| P5 | Simulation Engine 深度接入 | 支持项目执行计划、主线节点、多虚拟成员、事件驱动模拟、容量评估和行为反馈闭环。 | API、事件流、`/admin/simulation-lab/`、`/observer/` |
| P6 | 开源和生产化 | 支持多人协作、部署、权限、安全、可观测性和公开贡献流程。 | 全部入口 |

## P0：权威原型

### 目标

建立 Live OS 的权威数据骨架，证明真实用户和 Simulation Engine 未来可以围绕同一套 API 工作。

### 当前已覆盖

- 成员、任务、资源、积分账本、事件、申诉、规则版本、容量评估模型。
- 本地 MySQL 开发配置。
- 契约仓库 `big-apple-contracts` 和 Live OS 独立仓库边界。
- Django Admin 中文化。
- 幂等 seed 数据。
- `smoke_workflow` API 闭环。
- `/observer/` 只读观察台第一版。
- `start.bat` 和 `stop.bat` 本地启停脚本。
- `docs/ROADMAP.md` 中远期路线图。

### 完成标准

- `python manage.py check` 通过。
- `python manage.py makemigrations --check --dry-run` 通过。
- `python manage.py test core live_os observer workspace simulation simulation_lab --settings=live_os.test_settings` 通过。
- `python manage.py smoke_workflow` 可以在本地数据库跑通。
- `/admin/` 可用，中文界面正常加载 CSS 和 JS。
- `/observer/` 可以展示 seed 数据。
- README 和 `docs/` 说明当前能力和限制。

## P1：内部维护后台强化

### 目标

继续使用 Django Admin 服务开发期和早期运营，但明确它是内部维护后台，不是最终运营后台。

### 用户

- 项目维护者
- 早期管理员
- AI 开发助手
- 测试人员

### 能力

- 为核心模型补齐列表列、搜索、筛选、排序和只读字段。
- 为危险字段增加只读保护，避免误改账本、事件和历史状态。
- 为任务、成员、资源、申诉提供清晰的 Admin 分组和中文说明。
- 为 seed 数据补充更接近完整体场景的样本，包括异常、申诉、资源紧张和容量评估。
- 增加 Admin 操作说明文档，写清楚哪些字段可编辑、哪些只读、哪些只能通过 service 或 API 变化。

### 数据和契约

- 不优先新增外部 API。
- 如发现 contracts 缺少必要状态或字段，先修改 `big-apple-contracts`，再同步 Live OS。
- 账本和事件继续保持追加式语义。

### 完成标准

- 管理员可以通过 `/admin/` 快速定位成员、任务、资源、申诉和事件。
- 高风险数据默认不可直接改写，或文档明确说明风险。
- seed 数据能覆盖至少一个完整任务闭环、一个申诉场景和一个资源状态变化场景。
- `docs/DATABASE_SCHEMA.md` 和 Admin 使用说明保持同步。

## P2：成员工作台

### 目标

让候选成员和正式成员拥有面向自己的产品入口，而不是通过 Admin 或底层 API 参与系统。

### 用户

- 候选成员
- 正式成员
- 未来移动端或 Web 客户端
- Simulation Engine 中的虚拟成员

### 能力

- 查看可领取任务。
- 领取任务。
- 提交劳动结果。
- 查看任务状态和验收结果。
- 查看个人贡献积分流水。
- 查看个人身份状态、角色、加入时间和最近事件。
- 查看与自己相关的申诉和处理结果。

### 数据和契约

- 明确成员身份认证和 actor 映射规则。
- 扩展任务查询、领取、提交、个人账本和个人事件 API。
- 在 contracts 中定义成员工作台相关响应 schema。
- 为个人页面引入最小权限约束，避免成员查看不属于自己的隐私数据。

### 完成标准

- 一个候选成员可以不进入 `/admin/`，完成查看任务、领取任务、提交劳动、查看结果。
- 真实成员和虚拟成员使用同一组状态变化 API。
- 每个状态变化都产生可审计事件。
- 自动化测试覆盖成员任务闭环。

## P3：运营后台

### 目标

建立真正面向管理员的运营后台。它按业务流程组织，而不是按数据库表组织。

### 用户

- 管理员
- 运营人员
- 项目维护者

### 能力

- 成员运营：查看身份、贡献、风险提示、最近事件和备注。
- 任务运营：创建任务、审核任务、开放领取、指派、验收、关闭。
- 资源运营：查看库存、记录资源变动、关联任务和事件。
- 申诉处理：查看上下文、记录处理意见、形成处理结果。
- 事件审计：按成员、任务、资源、申诉过滤关键事件。
- 运营仪表盘：展示待处理任务、未关闭申诉、资源风险和近期异常。

### 数据和契约

- 运营后台可以先使用服务端渲染，也可以在需要交互时引入 HTMX 或独立前端。
- 所有写操作应走对应领域服务模块，不能直接在 view 中拼接状态变化；不要再新增 `core.services` 这种大杂烩服务门面。
- 需要逐步引入操作日志或审计模型，记录操作者、动作、对象、时间和原因。

### 完成标准

- 管理员可以完成任务发布到验收的主要流程，不需要直接编辑多张底层表。
- 危险操作需要明确确认、原因记录或权限约束。
- 运营后台和 Django Admin 的定位在文档中清晰区分。
- 关键流程有自动化测试和手工验收说明。

## P4：治理后台

### 目标

支持社区治理成员处理高影响规则、争议和复核流程。

### 用户

- 治理成员
- 管理员
- 审计者

### 能力

- 查看规则版本历史。
- 准备规则变更草案。
- 发布规则版本并记录生效时间。
- 处理高影响申诉和争议。
- 对积分、身份、资源和任务验收相关争议形成裁决记录。
- 查看关键操作审计链。
- 标记需要人工复核的异常事件。

### 数据和契约

- 规则版本需要更完整的结构化字段，包括版本号、生效状态、发布者、说明和变更摘要。
- 争议和裁决应有单独的状态流转和责任人记录。
- 高影响操作必须能追溯到具体实名责任人，不能只记录算法判断。

### 完成标准

- 治理成员可以查看规则版本和相关历史事件。
- 一条申诉可以从提交、受理、处理、裁决到关闭完整留痕。
- 高影响操作具备责任人、原因、时间和对象记录。
- 文档说明治理后台和运营后台的权限差异。

## P5：Simulation Engine 深度接入

### 目标

让 Simulation Engine 成为真实的外部客户端，通过 Live OS API 驱动多虚拟成员行为，并从事件流中接收反馈。

### 用户

- Simulation Engine
- AI 代理
- 项目维护者
- 观察者

### 能力

- 创建或同步虚拟成员身份。
- 获取开放任务、资源状态、规则版本和事件流。
- 以虚拟成员身份提交任务行为。
- 接收任务验收、积分、申诉、资源变化和容量评估反馈。
- 支持多虚拟成员并发行为。
- 支持模拟日期、模拟批次和实验标识。
- 将主线任务线数据库化，支持计划版本、树状节点、依赖、预算、人力、工期、完成标准和容量影响。
- 在 `/admin/simulation-lab/` 管理仿真实验启动和推进，在固定 world 的 `/observer/` 展示世界日志、运行结果和复盘状态。
- 支持自动模拟按主线计划推进到失败，并记录失败、节点状态和计划修订建议。
- 支持把计划修订建议转化为结构化计划变更集和变更操作。

### 数据和契约

- 明确虚拟成员和真实成员共享哪些字段，隔离哪些字段。
- 为模拟批次、模拟日期、行为来源和实验元数据建立结构化记录。
- 为项目执行计划建立结构化记录，模拟运行不能直接污染计划源头，只能产出修订建议。
- 为模拟运行、节点模拟状态、失败记录和修订建议建立结构化记录。
- 为结构化计划变更集和变更操作建立记录，作为后续生成新计划版本的输入。
- 事件流需要支持客户端增量读取。
- 容量评估需要从占位字段演进为可追踪、可复现的数据结构。

### 完成标准

- Simulation Engine 不需要数据库权限即可跑通多虚拟成员任务闭环。
- Live OS 能区分真实成员和虚拟成员的来源，但不破坏统一 API。
- 观察台能展示模拟运行对系统状态的影响。
- 观察台能展示当前执行计划主线、当前节点、必要节点完成数、成本和容量影响。
- 仿真推进可以在页面上被观察、暂停、继续和复盘，命令行只保留为开发自检工具。
- 自动模拟失败后可以生成待审核的计划修订建议，并且不会直接污染当前计划版本。
- 计划修订建议可以拆解为待审核的数据 patch 操作。
- 契约、示例和测试覆盖虚拟成员主要行为。

## P6：开源和生产化

### 目标

把项目从本地原型推进到多人协作、可部署、可审计、可持续维护的开源项目。

### 用户

- 开源贡献者
- 部署维护者
- 安全审查者
- 外部集成方

### 能力

- 认证和权限体系。
- 部署配置和环境变量文档。
- 数据库迁移和备份策略。
- 日志、指标和健康检查。
- CI 测试和契约校验。
- 贡献指南、代码规范、Issue 模板和 PR 模板。
- 安全边界和隐私数据处理说明。

### 数据和契约

- contracts 仓库成为外部集成的稳定入口。
- API 变更需要版本化策略。
- 数据库迁移需要兼容已有部署。
- 示例数据和演示环境不能包含真实隐私数据。

### 完成标准

- 新贡献者可以按文档在本地跑起系统。
- CI 能自动验证测试、迁移、格式和契约。
- 部署文档覆盖开发、测试和生产差异。
- 权限、审计和备份策略有明确文档。

## 横向能力路线

### 认证和权限

- P0-P1：本地开发和 Admin 超级用户。
- P2：成员身份和个人数据权限。
- P3：管理员权限分层。
- P4：治理成员权限和高影响操作复核。
- P6：生产级认证、会话、安全配置和审计。

### 审计和事件

- P0：基础事件流。
- P1：明确哪些数据只能追加或只读。
- P3：运营操作日志。
- P4：治理裁决和复核审计链。
- P6：生产级日志、导出和长期保存策略。

### 指标和观察台

- P0：观察台基础统计。
- P2：成员个人贡献和状态指标。
- P3：运营待办、风险和异常指标。
- P5：模拟批次、容量评估和系统反馈指标。
- P6：生产监控、健康检查和公开状态页。

### 测试和质量

- P0：模型、API 闭环和 smoke workflow。
- P1：Admin 配置和 seed 数据稳定性。
- P2：成员工作台流程测试。
- P3：运营流程测试。
- P4：治理流程和审计测试。
- P6：CI、契约校验、迁移检查和安全检查。
